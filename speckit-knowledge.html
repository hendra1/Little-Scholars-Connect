The Detailed Process

1

Initialize & Establish Principles

specify init

Set up the project structure and define the "Constitution". This governance guides all AI decisions regarding code quality, testing, and consistency.

/speckit.constitution
2

Create Specification

/speckit.specify

Describe WHAT you want to build and WHY. Do not focus on the tech stack yet. The AI creates a feature branch and structured spec document.

  • Focus on user stories
  • Define measurable success criteria
  • Generates specs/001-feature/spec.md
3

Refine & Clarify

/speckit.clarify

Run a structured clarification workflow. The AI quizzes you on edge cases and ambiguities to reduce rework downstream. Required before planning.

4

Technical Planning

/speckit.plan

Define the HOW. Provide your tech stack, architecture choices, and libraries. The AI maps requirements to technical decisions.

Research Agents: The AI may spin up research tasks to investigate library compatibility or specific implementation details before finalizing the plan.
5

Validation

Manual Audit

Ask the AI to audit its own plan. Check for missing pieces, over-engineering, or violations of the Constitution.

6

Task Breakdown

/speckit.tasks

Generate an actionable tasks.md file. This organizes work by user story, manages dependencies, and identifies tasks that can run in parallel [P].

7

Execute Implementation

/speckit.implement

The AI executes the tasks in order.

TDD Enforced Dependency Aware Auto-Refinement

The Constitution: Core Articles

Article I: Library-First

Every feature must begin as a standalone library. No exceptions. Forces modular design and clean boundaries.

Article II: CLI Interface

Every library must expose functionality via CLI (stdin/stdout). Ensures observability and easy testing.

Article III: Test-First

NON-NEGOTIABLE. No implementation code before tests are written, validated, and failing (Red phase).
